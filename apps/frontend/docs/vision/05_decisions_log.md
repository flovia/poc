# 意思決定ログ

> 最終更新: 2026-04-28
> 目的: 後から「なぜこう決めた?」を遡れるようにする

## D1: PoC のゴール

**結論**: 投資家/顧客デモ用のリッチ版

| 選択肢 | 採否 | 理由 |
|---|---|---|
| 同期会で見せる static モック | ✗ | 見栄え不足 |
| 実データで動く動作版 | ✗ | バックエンド準備が間に合わない |
| **投資家/顧客デモ用のリッチ版** | ○ | 見栄えとストーリーで判断材料を提供 |

## D2: 利用者

**結論**: 顧客 API Provider 向け

| 選択肢 | 採否 | 理由 |
|---|---|---|
| 社内 PM | ✗ | デモ価値が低い |
| **顧客 API Provider** | ○ | "今すぐ使いたい" 価値を訴求できる |
| 両対応 | ✗ | 焦点がぼやける |

## D3: ナラティブ

**結論**: メイン = Co-usage Discovery, サブ = Retention

| 選択肢 | 採否 | 理由 |
|---|---|---|
| 案A: Revenue Discovery | ✗→ベース | 一般的すぎ。Stripe Dashboard でもできる |
| 案B: Retention | ✗→サブ | 単独だと顧客の "今すぐ欲しい" 感が薄い |
| 案C: Live Ops | ✗ | 業務で毎日見る感が弱い |
| **Co-usage Discovery (新案)** | ○ | x402 公開データならではのユニーク機能。Stripe Dashboard では絶対できない |
| Co-usage + Retention サブ | ○ | 顧客価値 + 投資家ナラティブを両立 |

## D4: 視点 (競合 vs 併用)

**結論**: 併用パターン分析

| 選択肢 | 採否 | 理由 |
|---|---|---|
| 競合分析 | ✗ | 守りの分析。他社 API は競合とは限らない |
| **併用パターン分析** | ○ | 攻めの分析。エージェントのワークフロー全体が見える本質的な視点 |

## D5: 主役ビジュアル

**結論**: メイン=ネットワーク図 / サブ=Sankey / 戦略示唆=バブル

| 可視化 | 配置 | 理由 |
|---|---|---|
| **ネットワーク図** | 画面2 メイン | デモ映え最強, エコシステム構造が見える |
| **Sankey** | 画面2 下部 | ワークフローの流れ・順序が見える |
| **バブル** | 画面3 メイン | 戦略示唆 (右上=提携優先) を出せる |

## D6: データ粒度

**結論**: Provider 単位 / Provider × API パス単位 の両対応

| 選択肢 | 採否 | 理由 |
|---|---|---|
| Provider 単位のみ | ✗ | 顧客によってはパス単位でアドレスが分かれているため不十分 |
| Provider × API パスのみ | ✗ | パス単位アドレスがない顧客に対応できない |
| **両対応** | ○ | 顧客側の x402 設定の差異を吸収 |

## D7: 画面3 (Co-usage Patterns) の MVP 含有

**結論**: MVP に含める

| 選択肢 | 採否 | 理由 |
|---|---|---|
| 含めない (画面1+2 のみ) | ✗ | 経営層向けの戦略示唆ビューが欠ける |
| **含める** | ○ | デモストーリーが 3 段階 (個別→詳細→全体集約) で完結する |

## D8: 技術スタック

**結論**: Next.js (App Router) + pnpm + Vercel + モック JSON

理由:
- Vercel デプロイで URL 共有が容易 → 投資家/顧客に見せやすい
- App Router で Server Components が使える (モック JSON 配信に好適)
- pnpm はユーザー指定

## D9: アクセスモデル

**結論**: 完全フラット (認証なし)

| 選択肢 | 採否 | 理由 |
|---|---|---|
| **完全フラット** | ○ | PoC スコープに収まる。公開ディレクトリ的位置付け |
| View as 切替モデル | ✗ | PoC では過剰。将来認証導入時に再検討 |
| 自社/他社モード | ✗ | 認証が前提となり PoC 範囲外 |
| タブ型 | ✗ | UI が重い |

将来 SDK 経由の自社専用データ表示時に認証 + モード概念を追加する想定。
PoC 段階で **公開データ / 自社専用データのコンポーネント分離**を意識しておく。

## D10: pay_to の保存場所

**結論**: ブラウザ側 (localStorage)

| 選択肢 | 採否 | 理由 |
|---|---|---|
| **localStorage** | ○ | 認証なしで完結。切替・削除が容易 |
| URL クエリパラメータのみ | ✗ | 永続化されない / 共有時に他人に渡る |
| サーバー側保存 | ✗ | 認証が必要になる |

### 補助仕様

- サイドバー / ヘッダーの Provider セレクターで簡単切替
- 各エントリに削除ボタン
- Simple モード (pay_to 1 つ) / Advanced モード (API パスごとに複数 pay_to) の両対応

## D11: Sankey の採用見送り

**結論**: PoC では Sankey 不採用、Activity Timeline (時系列リスト) で代替

理由:
- 「意図解釈はせず、どの順番で・どの x402 リクエストをいつ使ったかを把握する」方針 (ユーザー指定)
- Activity Timeline で素データを見せる方が PoC スコープに合う
- Sankey は Phase 2 で「ワークフロー集約ビュー」として再検討の余地あり

## D12: Wallet 360° の主役の格上げ

**結論**: Activity Timeline を主役、Co-usage Map は補助

理由:
- 当初案では Co-usage Map が主役だったが、上記 D11 と整合する形で時系列リストを格上げ
- 時系列で素データが見える方が "信頼感" のあるデモになる
- Co-usage Map は構造的補強の役割に

## D14: Wallet 360° の主役 (確定)

**結論**: Activity Timeline を主役で固定 (Co-usage Map は補強役)

| 選択肢 | 採否 | 理由 |
|---|---|---|
| **Activity Timeline 主役** | ○ | デモ脚本のクライマックス a (理解の獲得) は Timeline でしか成立しない |
| Co-usage Map 主役 | ✗ | 視覚インパクトはあるが、売るべき価値 (顧客ワークフロー理解 + 売上機会発見) を直接示せない |

詳細: [10_design_review.md](10_design_review.md)

## D15: デモ脚本に基づくクリティカル UI 5 要素の追加

**結論**: 以下 5 要素を Wallet 360° に追加

1. Free Tier Progress Bar
2. 7d Volume Sparkline
3. Upsell Opportunity Card
4. Workflow Summary Strip
5. Entry-point Badge

理由: デモのクライマックス a (理解の獲得) と b (売上機会発見) を成立させるため必須。
詳細: [10_design_review.md](10_design_review.md)

## D13: デザイン方向

**結論**: ハイブリッド (UI shell = Precision Graph / 主役画面 = Ambient Mesh)

| 選択肢 | 採否 | 理由 |
|---|---|---|
| 案1 単独 (Precision Graph) | ✗ | Flovia の独自性が薄い |
| 案2 単独 (Editorial Signal) | ✗ | 技術感が弱い |
| 案3 単独 (Ambient Mesh) | ✗ | 演出過多で crypto 風に振れるリスク + 実装難度高 |
| **ハイブリッド (案1 + 案3)** | ○ | 定型 UI は端正に、主役画面のみ映えさせる。コスト/効果バランス最良 |

詳細は [06_design_direction.md](06_design_direction.md)。
ベース配色 = Precision Graph (Slate/Teal)、主役画面のみ Ambient Mesh アクセント (深ネイビー + 青発光) を重ねる。
ダークモードを**デモ既定**とする。
